# 心动放送现阶段报告

更新时间：2026-05-30  
范围：前端仓库内 `LoveShow / 心动放送 / 唯一心动线` 现有实现、内容资产、交互链路与策划可延展方向。

## 1. 一句话定位

`心动放送` 当前已经不是单一的“角色约会聊天”功能，而是一个以用户为唯一恋爱主轴的 AI 恋综玩法雏形：用户以本季主角身份进入《唯一心动线》，选择或确认嘉宾阵容，在节目组通知、镜头调度、小手机私聊、隐藏心令、心动风向、片段剧场、热榜舆论和回忆沉淀之间推进关系。

核心差异点在于：它把常规角色聊天包装成“综艺录制现场”，用节目组机制制造选择压力、群像反应、观众误读和二次传播，而不是让用户只和单个角色线性对话。

## 2. 当前可见入口与命名体系

- 应用入口已注册为 `AppID.LoveShow = 'love_show'`，启动器显示名为 `心动放送`。
- 正式节目名为 `唯一心动线`。
- 主要前台术语已经统一：
  - `心动放送`：应用名。
  - `唯一心动线`：本季节目名。
  - `镜头之外`：悬浮小手机私聊入口。
  - `放送通知`：节目组选择点/任务提示。
  - `隐藏心令`：用户专属任务。
  - `心动风向`、`风向揭晓`、`心动热榜`：观众舆论与下一轮镜头倾向。
  - `心动片段`、`心动回声`：由风向触发的单独/三人片段及其沉淀结果。

这些命名已经具备策划包装感，可以作为方案的第一版栏目体系。

## 3. 当前产品体验链路

### 3.1 入场：开播前选角

用户进入 `心动放送` 后，如果当前没有确认过本季阵容，会先进入选角页。系统默认把当前聊天角色纳入本季，也允许用户锁定其他角色库角色。阵容上限为 4 位正式嘉宾。

当真实角色不足 4 位时，系统会补齐“节目组邀请嘉宾”。这类 NPC 不会写入角色库，而是作为 LoveShow 内部运行态嘉宾存在。系统已有 3 个本地兜底 NPC 模板，并支持通过副 API 生成更贴合现有阵容的 NPC。

策划意义：这一层已经支持“用户自选男嘉宾 + 节目组空降男嘉宾”的节目结构，可以包装成开播前 casting、嘉宾官宣、空降位、观察员推荐位。

### 3.2 开场：Day 1 初见片段

当前赛季从 Day 1 开始，默认会触发全体嘉宾客厅集合的 `group_event`。系统会自动生成初见场景，并通过导演镜头卡控制本轮在场嘉宾、镜头焦点、发言人、反应位和用户位置。

如果主 API 不可用，前端有硬编码兜底开场，确保页面仍能进入节目现场。

策划意义：Day 1 可以做成强仪式感开场，例如“全员初见”“第一眼心动镜头”“节目组宣读规则”“第一张约会邀请卡”。

### 3.3 正片：即时场景对话

正片由 `LoveShowScene` 渲染，支持：

- 直播状态标识。
- Day 与地点标题。
- 当前场景氛围。
- 在场嘉宾 strip。
- AI 输出脚本解析：旁白、角色对白、单采、手机消息、普通文本。
- 用户输入。
- “收束片段”按钮，用于手动结束当前小场景并进入幕后结算。

AI 输出不是直接整段聊天气泡，而会被解析为更像综艺剧本的结构化节点，例如：

- 星号包裹的动作/环境描写。
- `角色名：台词` 的对白。
- 对镜头单采。
- 手机消息。

策划意义：这里可以承载“正片剪辑感”，不只是聊天，而是现场、镜头、停顿、反应、单采混合的观感。

### 3.4 幕后小手机：镜头之外

页面右下有悬浮手机入口，当前包含 5 个 tab：

- `镜头之外`：和嘉宾私聊。
- `放送通知`：当前选择点。
- `隐藏心令`：用户任务列表，可标记完成。
- `心动档案`：嘉宾状态、好感、印象卡摘要。
- `心动热榜`：心动风向与微博/小红书式观众讨论。

小手机本身支持：

- 拖拽位置。
- 未读红点。
- 自定义壁纸上传。
- 恢复默认壁纸。
- 按 tab 标记未读。

策划意义：这个小手机是恋综玩法的中枢。它可以被包装为“节目组发放的录制手机”，承载暗线、舆论、私聊、任务、嘉宾档案，后续也适合加“短信投递”“观察室权限”“观众投票”。

### 3.5 节目组选择点

当前 `ChoicePoint` 类型包括：

- `group_event`：集体活动，Day 1 强制触发。
- `date_card`：约会券给谁，必选。
- `sms_target`：匿名短信发给谁，可选。
- `sms_content`：短信内容自由输入。
- `daily_mission`：隐藏心令，接受或稍后。
- 约会卡外出地点：由节目组按赛季、日期和节点稳定安排，不再让用户单独选择合宿屋地点。
- `observatory`：观察室/查看独白。
- `interview`：单采间自由表达。

实际选择生成逻辑仍偏 V0：按 day、phase 和已完成选择历史决定下一个选择点；目前没有完整节目日程表，也没有真正的淘汰、终选闭环。

策划意义：选择点结构已经够用来做第一版互动剧本框架，但还需要策划补“每日节奏表”和“节点规则”，否则体验会更像连续即兴片段，而不是一季完整节目。

### 3.6 收束结算：状态、印象、回忆

用户点击“收束”后，系统会：

1. 生成或兜底生成场景摘要。
2. 保存回忆卡。
3. 对当前在场嘉宾逐个更新角色状态。
4. 更新嘉宾对用户的印象卡。
5. 生成下一张放送通知。
6. 生成心动风向。
7. 可能触发心动片段票根。
8. 清空当前 transcript，进入下一场景或等待用户选择。

角色状态卡字段包括：

- 好感度 0-100。
- 心情：期待、吃醋、受伤、心动、试探、冷淡、紧张、开心。
- 自信度 0-100。
- 策略：主动进攻、欲擒故纵、默默守护、直球表白、观望、撤退。
- 嫉妒对象。
- 内心想法。
- 最近更新场景。

印象卡字段包括：

- 感知到的用户特质。
- 已知事实。
- 暂时理解。
- 一句整体印象。
- 历史快照。

策划意义：这是“关系养成”的底层。它可以支撑嘉宾差异化反应、热榜误读、单采内容、终选判断和复盘卡片。

## 4. AI 架构现状

### 4.1 主 API：负责演出

主 API 负责生成自然语言正片内容。当前分两种模式：

- 场景模式：用多嘉宾 preamble + 当前场景上下文 + DirectorBeat 镜头卡，让模型演当前一小拍。
- 私聊模式：用单嘉宾平行世界上下文 + LoveShow preamble，让模型只回复镜头之外私聊。

系统明确要求：

- 所有嘉宾都是正式嘉宾。
- 用户是唯一恋爱主轴。
- 嘉宾之间可以竞争、观察、误解、助攻，但不能发展成主 CP。
- 一次只演当前小拍。
- 不替用户做选择。
- 只让 DirectorBeat 安排的 1-3 位嘉宾明显发言。

### 4.2 副 API：负责幕后结构化判断

副 API 当前承担：

- 生成 DirectorBeat 导演镜头卡。
- 生成/展开 NPC 人设。
- 生成场景摘要。
- 评估角色状态。
- 更新印象卡。
- 生成心动热榜。
- 生成隐藏心令。

所有副 API 输出都要求 JSON 或短文本，且有若干本地 fallback。

### 4.3 安全与方向控制

当前实现反复强调“用户中心向”，并且在多个模块过滤风险表达：

- 禁止嘉宾互相心动、嘉宾 CP、互选、锁死、谁和谁最配。
- 心动风向必须围绕用户与嘉宾。
- 心动片段不能替用户决定心动归属。
- 印象卡避免攻略女性、审判女性、物化女性、霸总修罗场式措辞。
- 对危险词和高风险印象表达有修正/兜底逻辑。

策划意义：这一点非常关键。当前产品定位不是“男嘉宾互嗑 CP 的群像恋综”，而是“用户作为唯一主角，被多位正式嘉宾围绕”的沉浸式恋综。

## 8. 当前主要缺口与风险

### 8.1 一季节目闭环还不完整

类型层定义了 Day 1-5、finale、completed、eliminations、finalChoice，但现有流程主要停留在“不断生成下一张选择点 + 收束进入下一片段”。目前没有成熟的：

- 每日固定节目结构。
- Day 结束推进。
- Day 5 终选。
- 嘉宾告白/拒绝/开放式结局。
- 淘汰规则或明确“不淘汰”的节目规则。
- 赛季完成后的总结页。

策划需要先决定：这档节目到底是 5 天强结构恋综，还是无限流式“恋综片段生成器”。当前代码更接近后者，但类型设计给前者留了入口。

### 8.2 选择点调度还是启发式

`generateNextChoicePoint` 使用 day、phase、completedChoiceIds 做基础判断，但 phase 推进较弱，戏份平衡也只是简化计数。容易出现体验上“通知很多，但节目节奏不够像真正综艺”的问题。

建议策划提供 Day 1-5 的固定栏目表，再让代码做轻量状态机，而不是完全即兴。

### 8.3 主 API 调用未接入统一 API ledger

仓库有 `apiRequestLedger`，但 LoveShowApp 当前直接 fetch 主 API/副 API，未接入统一追踪。设置页里 `loveshow` feature 当前显示为“摘星楼”，且摘星楼 API 也使用了 `feature: 'loveshow'`，这会导致用量/错误归因混乱。

这不影响玩法方案，但会影响后续测试、成本统计和线上问题排查。

### 8.4 UI 已有雏形，但需要真机/浏览器走查

当前 CSS 与组件结构已较复杂，包含小手机、浮层、票根、热榜、状态 rail 等。报告阶段只做代码阅读，尚未确认：

- 移动端小屏是否拥挤。
- 小手机拖拽、上传壁纸、tab 切换是否有遮挡。
- 心动片段票根是否在各种屏幕高度下可完整操作。
- 地点图片是否全部加载成功。
- 长文案是否溢出。

### 8.5 心动片段剧场目前偏“结果沉淀”，不是完整可演片段

当前剧场票根可以选择嘉宾和地点，开始/完成片段后生成回声、记忆、热帖，并把余波注入下一场氛围。但“片段内部”还不是一个独立的可对话演出空间，更像是快速生成结果。

如果策划希望它成为重点玩法，需要决定：

- 片段是自动生成一段短剧情，还是进入一个可聊 1-3 轮的小剧场。
- solo 和 triangle 的差异化交互是什么。
- 片段完成后如何明确影响嘉宾状态、热榜和后续选择。

### 8.6 观察室/单采潜力尚未完全释放

类型和 prompt 中已经有 `observatory`、`interview`、`confession_room`，但目前没有做成强 UI 栏目。单采目前更多是脚本格式和选择点概念，观察室也没有形成“偷看某位嘉宾内心/观众解读/主持人点评”的完整体验。

### 8.7 NPC 与角色库边界需要产品决策

NPC 当前不会保存进角色库，这是合理的轻量方案。但策划需要决定：

- 用户能否把节目组嘉宾“转正”为角色库角色。
- 赛季结束后 NPC 是否可复用到下一季。
- NPC 是否支持头像生成/选择。
- NPC 是否拥有独立记忆中心，还是只保留 LoveShow 内部记忆。

## 9. 策划可直接展开的方案方向

### 方向 A：5 天季播恋综

适合把 `SeasonState.day = 1-5` 做实。建议结构：

- Day 1：全员初见、第一印象、首次约会券、匿名短信。
- Day 2：分组任务、第一次心动风向、单采开放。
- Day 3：空降/反转、三人片段、隐藏心令升级。
- Day 4：未公开片段放映、观察室权限、关系压力测试。
- Day 5：最终约会、嘉宾单采告白、用户终选、季终回忆卡。

优点：策划感强，便于做宣传和留存。  
代价：需要补状态机、Day 推进和终选页。

### 方向 B：无限片段恋综

弱化 Day 1-5，主打“随时打开一段心动放送”。每次用户收束片段后，系统继续生成风向、票根、通知、私聊。最终不强调结局，只强调沉浸和反复生成。

优点：开发成本低，贴合当前代码。  
代价：缺少明确目标，长期体验可能散。

### 方向 C：小手机驱动的幕后恋综

把“镜头之外”做成核心：节目现场只是正片，小手机才是暗线。重点强化：

- 私聊短信。
- 心令任务。
- 热榜舆论。
- 观察室偷看。
- 嘉宾档案变化。
- 节目组突发通知。

优点：和现有小手机实现高度匹配，差异化明显。  
代价：需要大量精细文案和通知节奏。

### 方向 D：片段剧场卡池

把心动片段做成“票根抽取/触发”的卡池系统。用户在风向后获得票根，选择嘉宾和地点，进入一段短互动，结束后得到回声、热帖和状态变化。

优点：适合做资产化地点、名场面收集、回忆卡。  
代价：需要把目前“快速完成结果”升级成真正片段体验。

## 10. 推荐下一阶段优先级

### P0：先定节目骨架

策划需要先拍板：

- 是 5 天季播，还是无限片段。
- 是否有最终选择。
- 是否存在淘汰；如果没有，要明确节目规则是“全员正式嘉宾，无淘汰”。
- 单季最多几位嘉宾，当前代码默认 4 位。
- 真实角色和节目组 NPC 的比例。

### P1：补 Day 节奏表

建议给每一天固定 3-5 个栏目，代码按栏目生成 ChoicePoint，而不是继续只靠启发式判断。最低可先做：

- 今日开场。
- 集体任务。
- 单独约会。
- 镜头之外短信。
- 单采/观察室。
- 心动风向。
- 今日收束。

### P1：做赛季终点

需要一个明确的终点页：

- 用户最终选择谁。
- 被选嘉宾回应。
- 未被选嘉宾单采。
- 本季名场面回顾。
- 心动曲线/好感变化。
- 可保存的季终卡。

### P2：把观察室做成栏目

观察室可以成为非常有恋综感的功能：

- 偷看某位嘉宾单采。
- 查看观众热议误读。
- 主持人口播式复盘。
- 解锁“未公开镜头”。
- 给下一轮选择提供软提示，但不替用户决定。

### P2：强化心动片段

把票根从“生成结果”升级为短流程：

1. 选择嘉宾和地点。
2. 进入专属片段场景。
3. 用户互动 1-3 轮。
4. 收束为心动回声。
5. 影响下一场镜头、热榜和嘉宾状态。

### P2：补可视化反馈

可加：

- 嘉宾好感/心情变化动效。
- 今日风向卡。
- 名场面截图卡。
- 嘉宾档案更新提示。
- 心令完成奖励弹窗。

## 11. 给策划的关键判断题

1. `心动放送` 是“恋综季播游戏”还是“恋综氛围聊天模式”？
2. 用户是否必须最终选择一位嘉宾？
3. 节目是否允许失败、拒绝、开放式结局？
4. 是否坚持“无嘉宾 CP，所有张力围绕用户”？
5. 空降 NPC 是常驻卖点，还是只在角色不足时补位？
6. 心动风向是纯舆论展示，还是会强影响下一场剧情？
7. 片段剧场是轻量结果卡，还是核心可玩模式？
8. 小手机是否作为全局中枢长期保留？

## 12. 现阶段结论

当前 LoveShow 已经具备一个“可玩的恋综系统骨架”：入口、选角、嘉宾补位、场景演出、小手机、私聊、通知、任务、热榜、风向、票根、回声、记忆和本地持久化都有了。它最强的部分是“用户中心向恋综”的规则意识和氛围包装；最弱的部分是“完整一季节目”的节奏闭环还没有完全落地。

如果现在要给策划做方案，建议不要从“再加几个功能”开始，而是先把节目形态定下来：一旦确定是 5 天季播或无限片段，现有模块都能很快归位。否则继续开发会容易堆出很多好看的组件，但用户不知道自己正在经历一季节目的哪一段。
